Lesson 2

Boot flow: từ reset tới app_main()

Khi cấp nguồn, CPU không tự nhiên nhảy vào code app của bạn. ESP32-S3 có ROM bootloader nằm cố định trong chip. ROM quyết định boot mode, load bootloader cấp hai từ SPI flash, rồi bootloader cấp hai chọn app partition và chạy app.

Bài học lý thuyết

Reset là điểm bắt đầu vật lý: nguồn vừa lên, nút reset được nhấn, watchdog kéo reset, hoặc phần mềm yêu cầu reboot. Sau reset, CPU phải biết chạy instruction đầu tiên ở đâu. Với ESP32-S3, instruction đầu tiên không nằm trong project của bạn, mà nằm trong ROM bootloader do Espressif ghi sẵn trong chip từ nhà máy.

ROM bootloader làm những việc cực sớm: đọc trạng thái boot mode, kiểm tra có vào download mode không, tìm flash, đọc image header, rồi load second-stage bootloader từ SPI flash. Second-stage bootloader là phần được ESP-IDF build ra cùng project. Nó biết partition table, flash mode, flash size, app slot nào đang active, OTA state có hợp lệ không, rồi mới load app firmware.

app_main() vì vậy không phải “dòng đầu tiên của thiết bị”. Nó chỉ là entry point của app sau khi ROM, bootloader, flash mapping, partition decision, một phần runtime ESP-IDF đã chuẩn bị xong. Nếu bootloader chọn sai partition, flash setting sai, hoặc chip vào download mode, app của bạn chưa bao giờ chạy tới app_main().

Khi annotate boot log, đừng đọc như log web server. Hãy chia theo phase. Dòng rất sớm thường thuộc ROM/bootloader. Dòng nói flash mode, clock, partition table, loaded app là bootloader/app loader. Dòng do code bạn in ra mới là app. Kỹ năng này giúp bạn biết bug nằm trước app hay trong app.

Cách nghĩ quan trọng nhất của lesson này: boot là một chuỗi bàn giao trách nhiệm. ROM chỉ biết đủ để tìm flash và nạp bootloader. Bootloader biết đủ để hiểu partition và chọn app. App chỉ bắt đầu khi môi trường tối thiểu đã sẵn sàng. Nếu thiết bị chết trước dòng log app đầu tiên, sửa app_main() thường không giải quyết gì.

Ví dụ đọc boot log bằng mắt firmware

Khi thấy một log kiểu rst:0x1, boot:0x8, SPIWP, mode:DIO, load:0x3f..., bạn chưa cần hiểu hết từng mã ngay. Việc đầu tiên là đánh dấu: đây là đoạn rất sớm, trước app. Sau đó tìm các dòng có tên bootloader/partition: chúng cho biết flash đang được đọc ra sao và app nào được chọn. Chỉ khi thấy log do project in ra, ví dụ Hello world hoặc tên component app, bạn mới bước vào vùng lỗi của application.

Hiểu sai thường gặp: tưởng reset là lúc app “restart function main”. Thật ra reset đưa chip về trạng thái gần như mới cấp nguồn. Toàn bộ boot chain chạy lại. Vì vậy reset loop có thể do nguồn, boot mode, flash, partition, bootloader, hoặc app. Lesson này dạy bạn tách năm khả năng đó trước khi đoán bug.

1. Reset Nguồn lên, reset button, watchdog, brownout hoặc software reset.
2. ROM bootloader Code cố định trong chip, đọc strapping pins và boot mode.
3. 2nd bootloader Nằm trong flash, do ESP-IDF build ra.
4. Partition table Biết app nằm ở đâu, NVS ở đâu, OTA slot nào active.
5. app_main() Code ứng dụng bắt đầu chạy, tạo task và init driver.

Boot mode

ESP32-S3 có thể boot từ flash hoặc vào download mode để flash firmware qua USB/UART. Trạng thái một số pin lúc reset quyết định boot mode. Vì vậy nút BOOT, RESET, USB serial, và esptool liên quan trực tiếp tới nhau.

Lab đọc boot log

# Sau khi cài ESP-IDF hoặc esptool/pyserial
idf.py monitor

# Hoặc dùng serial terminal nếu đã biết port
python -m serial.tools.miniterm /dev/cu.usbmodemXXXX 115200

Cách đọc boot log theo lớp

Reset causeTìm dòng cho biết reset vì power-on, software reset, watchdog, brownout, hay panic.
ROM phaseNhận ra đoạn rất sớm trước khi bootloader ESP-IDF in log app.
Bootloader phaseTìm flash mode, flash size, partition table, selected app partition.
App phaseTìm dòng đầu tiên do app của bạn log ra, thường sau khi scheduler/runtime sẵn sàng.
