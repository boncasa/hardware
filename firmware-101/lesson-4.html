Lesson 4

Memory model: ROM, flash, SRAM, IRAM, PSRAM

Khi đọc firmware, đừng chỉ hỏi “RAM còn bao nhiêu?”. Hãy hỏi vùng memory nào đang được dùng, vì không phải vùng nào cũng giống nhau về tốc độ, khả năng DMA, cache, và thời điểm truy cập.

Bài học lý thuyết

Trong app thường, “memory” hay được gom lại thành RAM. Trong firmware, bạn phải tách rõ storage lâu dài, RAM runtime, vùng code chạy nhanh, và vùng buffer ngoại vi. Cùng là bytes, nhưng đặt sai vùng có thể làm build fail, crash runtime, audio rè, LCD giật, hoặc interrupt chết khó hiểu.

ROM nằm trong chip và không sửa được. SPI flash lưu code/app/data lâu dài, nhưng truy cập qua cache và chậm hơn RAM. SRAM/DRAM là RAM trong chip, quý nhất cho heap, stack và data cần truy cập nhanh. IRAM là RAM instruction cho code cần chạy khi flash cache không sẵn sàng, ví dụ một số interrupt handler. PSRAM là RAM ngoài nếu module có, dung lượng lớn hơn nhưng chậm hơn và không phải mọi DMA/peripheral đều dùng trực tiếp được.

Khi build, linker đặt code/data vào các section. .text thường là code; .rodata là constant; .data là global/static có giá trị khởi tạo; .bss là global/static zero-initialized. Heap là vùng cấp phát runtime. Task stack là vùng stack riêng cho từng FreeRTOS task. Một local buffer lớn không ăn heap, nó ăn task stack. Một global array lớn có thể ăn DRAM hoặc BSS và làm app thiếu RAM ngay từ lúc boot.

DMA là điểm software engineer rất dễ bỏ qua. Một số peripheral như I2S/SPI có thể đọc/ghi memory trực tiếp mà không cần CPU copy từng byte. Nhưng DMA thường yêu cầu buffer nằm ở vùng memory phù hợp, alignment đúng, size đúng. Vì vậy “cứ malloc buffer lớn trong PSRAM” không luôn đúng, nhất là với audio/LCD path.

Một cách đọc memory cho firmware là luôn hỏi bốn câu: bytes này sống bao lâu, CPU hay peripheral đọc nó, nó cần nhanh tới mức nào, và nó có được dùng khi flash cache tạm tắt không. Constant text có thể nằm ở flash. Stack frame ngắn hạn nằm trong task stack. Audio buffer dài có thể cần PSRAM nhưng DMA path có thể bắt bạn dùng internal memory. Interrupt handler quan trọng có thể cần IRAM.

Ví dụ lỗi rất thật: bạn thêm một array local uint8_t buf[16384] trong audio task có stack 4096 bytes. Heap vẫn báo còn nhiều, nhưng task vẫn panic vì stack tràn. Hoặc bạn đưa LCD frame buffer vào PSRAM, build chạy, nhưng DMA không nhận buffer đó nên màn hình giật hoặc driver báo lỗi. Lesson này dạy bạn không dùng một chữ “RAM” cho mọi vùng nhớ.

Vùng Nó là gì Dùng cho Điểm dễ lỗi
ROM Code cố định trong chip ROM bootloader, hàm hệ thống thấp Không sửa được
SPI Flash Storage ngoài chip, non-volatile Bootloader, app, NVS, filesystem Chậm hơn RAM, phụ thuộc cache, erase/write theo block
SRAM/DRAM RAM trong chip Heap, stack, global data Ít, stack overflow rất hay gặp
IRAM RAM cho instruction Interrupt handler, code cần chạy khi flash cache off Đặt quá nhiều code vào IRAM sẽ hết nhanh
PSRAM RAM ngoài, nếu module có Buffer lớn: audio, image, UI, network Chậm hơn, không phải DMA nào cũng dùng được trực tiếp

Section trong binary

  • .text: code.
  • .rodata: constant/read-only data.
  • .data: global/static có giá trị khởi tạo.
  • .bss: global/static zero-initialized.
  • Heap: cấp phát runtime bằng allocator.
  • Task stack: stack riêng của từng FreeRTOS task.

Bài tập tư duy memory

  • Nếu tạo một array global 200 KB thì nó đi vào section nào? Điều gì xảy ra nếu internal RAM không đủ?
  • Nếu tạo local buffer 16 KB trong một task stack 4 KB thì lỗi có thể biểu hiện ra sao?
  • Vì sao interrupt handler thường không nên gọi function tùy tiện hoặc đụng flash khi cache bị tắt?
  • Vì sao audio/LCD buffer lớn có thể cần PSRAM, nhưng DMA path lại có yêu cầu riêng?
