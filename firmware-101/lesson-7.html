Lesson 7

Build artifacts: .elf, .bin, .map

Build firmware không chỉ tạo một file app. ESP-IDF tạo nhiều artifact, mỗi cái trả lời một câu hỏi khác nhau.

Bài học lý thuyết

Khi chạy idf.py build, toolchain compile source thành object files, link chúng thành một ELF, rồi tạo các binary image để flash. Nếu chỉ nhìn “build succeeded”, bạn bỏ lỡ phần quan trọng nhất: build đã tạo file nào, file đó ghi vào offset nào, và symbol nào đang dùng RAM/IRAM/flash.

.elf là file giàu thông tin nhất cho debug. Nó chứa symbol, address mapping và thông tin để decode backtrace. Khi panic log in ra PC/address, bạn cần ELF đúng build để biết address đó thuộc function nào. .bin là image được flash vào chip; nó phù hợp để nạp firmware, nhưng nghèo thông tin hơn cho debug.

.map là báo cáo của linker. Nó trả lời câu hỏi “ai đang ăn memory?”. Khi firmware tự nhiên vượt IRAM, DRAM giảm mạnh, hoặc binary phình ra, map file giúp tìm component/symbol lớn. idf.py size cho cái nhìn tổng quan; map file cho điều tra chi tiết.

Flash command cũng là tài liệu. Nó thường ghi bootloader vào một offset, partition table vào offset khác, app binary vào offset app. Đọc command flash giúp bạn nối Lesson 3 với build thật: file nào đi vào vùng nào của flash.

Build artifact là cách firmware để lại dấu vết kiểm chứng. Source code nói “mình muốn build gì”. Artifact nói “toolchain thật sự tạo ra gì”. Khi một thay đổi làm app lớn thêm 400 KB, build log và size output cho bạn biết đó là flash code, rodata, DRAM hay IRAM. Khi panic backtrace có địa chỉ, ELF cho bạn dịch địa chỉ thành function. Khi nghi partition sai, command flash cho bạn biết file nào đã đi vào offset nào.

Hiểu sai thường gặp: gửi cho người khác mỗi file .bin rồi hỏi decode crash. Nếu không có đúng .elf của build đó, backtrace mất phần lớn giá trị. Một firmware workflow tử tế lưu commit, sdkconfig, ELF, MAP, BIN và log cùng nhau. Lesson này dạy bạn giữ evidence để debug được sau này.

Artifact reading order

1. Build log What got built? Xem bootloader, partition table, app binary được tạo ở path nào. Đừng chỉ nhìn “build succeeded”.
2. Partition table Where will it flash? Đối chiếu offset trong command flash với partition table để hiểu từng file sẽ ghi vào vùng nào.
3. Size output How big is each section? Nhìn flash code/data, DRAM, IRAM. Khi một feature làm size tăng mạnh, quay lại đây.
4. Map file Who used the memory? Dùng khi muốn biết symbol/component cụ thể nào đang ăn RAM/IRAM/flash.
