Lesson 1

Firmware là gì?

Firmware là software chạy sát phần cứng. Nó khởi tạo chip, cấu hình pin, tạo task, nói chuyện với ngoại vi, lưu config, kết nối Wi-Fi, cập nhật màn hình, đọc microphone, phát speaker, và xử lý lỗi khi mọi thứ không như ý.

Bài học lý thuyết

Nếu backend/frontend là code chạy trên một môi trường đã được chuẩn bị sẵn, firmware là code chạy khi môi trường đó còn chưa hoàn chỉnh. Trên server, OS đã lo boot máy, chia process, cấp virtual memory, mount filesystem, quản lý network stack. Trên firmware, nhất là ESP32-S3, code của bạn sống gần hơn với câu hỏi “chip vừa reset, bây giờ phải bật gì lên để board hoạt động?”.

Một firmware thực tế thường có hai phần đan vào nhau. Phần thứ nhất là bring-up: khởi tạo clock, flash, RAM, pin, peripheral, driver, task scheduler, log, Wi-Fi, audio codec, màn hình. Phần thứ hai là application logic: xử lý nút bấm, nhận audio, phát speaker, update UI, gọi network API, lưu config, OTA update. Người mới hay nhảy thẳng vào application logic, nhưng khi lỗi xảy ra thì lại phải quay về bring-up.

Trên ESP32-S3, bạn không bắt đầu từ “app render màn hình” như frontend. Bạn bắt đầu từ một chuỗi lớp: ROM bootloader trong chip chạy trước, second-stage bootloader trong flash chạy tiếp, ESP-IDF runtime dựng hệ thống tối thiểu, rồi cuối cùng mới tới app_main(). Vì vậy app_main() là entry point của app firmware, không phải instruction đầu tiên sau khi cấp nguồn.

Từ vựng quan trọng nhất bài này là ROM, flash, RAM, peripheral, driver, và register. ROM là vùng code cố định trong chip, ví dụ ROM bootloader. Flash là nơi firmware image nằm lâu dài, giống ổ đĩa nhỏ gắn ngoài chip. RAM là nơi dữ liệu runtime sống khi firmware đang chạy. Peripheral là khối phần cứng trong chip như GPIO, UART, SPI, I2C, I2S, ADC, PWM. Register là các ô điều khiển phần cứng. Driver là code viết vào register đúng thứ tự để peripheral hoạt động theo ý mình.

Ví dụ khi bạn gọi API để vẽ lên LCD, firmware không chỉ “render UI”. Nó có thể đang cấp nguồn màn hình qua GPIO, kéo chân reset, cấu hình SPI bus, chọn clock, tạo DMA buffer, gửi command init sequence, rồi mới đẩy pixel. Nếu sai pin map thì màn hình đen. Nếu sai SPI mode thì hình rác. Nếu buffer nằm sai loại RAM thì DMA không đọc được. Nếu task render chiếm CPU quá lâu thì audio có thể giật hoặc watchdog reset.

Firmware crash cũng khác app crash. Backend chết thì process restart, log còn trên disk, request sau có thể chạy tiếp. Firmware chết có thể reset cả chip, mất gần hết context runtime, chỉ còn serial log, panic dump, reset reason, hoặc tệ hơn là boot loop. Vì vậy khi đọc firmware, luôn tự hỏi ba câu: code/data này nằm ở đâu, chạy lúc nào, và đang chạm phần cứng nào?

Đọc firmware theo lớp

Khi mở một firmware project, đừng đọc như một web app. Hãy đọc theo lớp từ dưới lên. Lớp thấp nhất là chip và board: chip gì, flash bao nhiêu, PSRAM có không, những pin nào nối tới màn hình, mic, speaker, button. Lớp kế là boot và memory: bootloader chọn app nào, partition table chia flash ra sao, RAM dùng cho heap/task/buffer thế nào. Lớp tiếp theo là driver: LCD qua SPI hay RGB, mic/speaker qua I2S hay codec riêng, touch/sensor qua I2C hay GPIO. Lớp trên cùng mới là app state, UI, protocol, cloud, OTA.

Mental model của Lesson 1: firmware là app logic cộng với trách nhiệm hiểu môi trường chạy của chính nó. Nếu chưa biết code đang ở lớp nào, rất dễ debug nhầm chỗ.

Câu hỏi cần nắm

  • Firmware image nằm lâu dài ở đâu, và khi chạy thì phần nào được load vào RAM?
  • Vì sao app_main() không phải code đầu tiên chạy sau reset?
  • Peripheral khác linh kiện ngoài board như thế nào?
  • Driver làm gì với register, pin, clock, interrupt, DMA?
  • Khi gọi driver LCD, mic, speaker, Wi-Fi, thật ra firmware đang chạm lớp phần cứng nào?
  • Vì sao firmware crash có thể thành reset loop thay vì chỉ chết một process?
